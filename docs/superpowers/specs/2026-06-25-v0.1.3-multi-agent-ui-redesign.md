# v0.1.3 — Multi-Agent UI Redesign

Date: 2026-06-25
Scope: Full-stack. Schema migration + API layer + frontend restructure.

## Problem

The current UI is a fixed two-panel layout (agent config + chat workbench) with a
single implicit "default" agent. There is no way to create multiple agents, and
the config/chat panels are tightly coupled. As the product grows, users need to:

1. Create and manage multiple agents with different configurations.
2. Browse chat history independently from agent configuration.
3. Have a clean, focused interface for each activity (chatting vs configuring).

## Design Decisions

### Sidebar: Two Collapsible Sections

The left sidebar splits into two accordion-style sections:

- **Agents** (upper): Lists all created agents. Each item shows agent name.
  Includes a "+ Create Agent" button and a "Start chat" action per agent.
  Note: agent names are not unique — if two agents share a name, they will
  appear as identical labels in the sidebar (known v0.1.3 limitation).
- **Chats** (lower): Lists all chat sessions. Each item shows title, associated
  agent name, relative timestamp, and last message preview. Includes a
  "+ New chat" button at the top.

Both sections default to expanded on page load. Clicking a specific item in
either section determines what the right-side area displays.

**State persistence:** The sidebar's expand/collapse state for each section and
the currently selected item (chat or agent ID) are persisted to `localStorage`.
On page reload, the sidebar restores which sections are expanded and which item
is selected (re-fetching its data from the API). If the persisted item no longer
exists, fall back to the empty/welcome state.

### Right-Side Views

The right-side area is a single-view region (not split-panel). The view switches
based on what is selected in the sidebar:

| Selection         | Right-side content                          |
|-------------------|---------------------------------------------|
| Nothing (default) | Empty state / welcome message               |
| A Chat session    | Full-width chat interface (messages + input) |
| An Agent          | Full-width agent configuration page         |

**Chat view:**
- Full-width message list + input box at the bottom.
- Top header displays "Chat with [Agent Name]" — agent name is clickable,
  navigates to that agent's config view.
- Input box belongs exclusively to Chat view (not shown on Agent config page).

**Agent config view:**
- Full-width configuration surface with existing tabs (Profile / Model / Tools).
- Save button persists to database.
- No embedded chat workbench (removed from this version).

### Landing State

When the application first loads, the right-side area shows an empty state with a
brief prompt: "Select a chat or agent from the sidebar." No item is pre-selected.

### Agent ↔ Chat Relationship

**Binding:** Each chat session is associated with one agent via `agent_id` FK.

**Live spec:** When sending a message, the system always fetches the agent's
*current* configuration from the `agents` table — not a frozen snapshot from
session creation time. This means:

- Changing an agent's model, system prompt, or tools affects all future messages
  in existing sessions bound to that agent.
- This is intentional for an agent-builder workflow where the user iterates on
  config and tests interactively.
- Historical messages retain their own `agent_spec_snapshot` at the task level
  for audit/debugging purposes.

**Snapshot capture timing:** The per-task `agent_spec_snapshot` on `agent_tasks`
is captured at task creation time — i.e., the moment the user sends a message
and the API handler creates the `agent_task` row. The flow:

1. Client sends `POST /api/chat-sessions/:id/messages` with `{ message, runtimeSecrets }` (no `agentSpec`).
2. API loads the session → reads `agent_id`.
3. API fetches the agent's *current* spec from the `agents` table.
4. API validates the spec.
5. API inserts the user `chat_message`, then inserts the `agent_task` with the
   just-fetched spec persisted into `agent_spec_snapshot`.
6. Runner executes against that snapshot.

Once a task exists, its snapshot is frozen for that task's lifetime. Subsequent
messages on the same session re-fetch the agent's then-current spec, so edits
made between messages take effect immediately on the next message.

**Deletion:** Agent deletion is NOT supported in v0.1.3. Only create and edit.

### Creating a New Chat

Two entry points:

1. **"+ New chat" button** (top of Chats section): Uses the agent with the most
   recent `updated_at` timestamp (or the first agent by `created_at` if none
   have been edited).
2. **"Start chat" per agent** (in Agents section): Binds the new chat to that
   specific agent.

**Lazy creation:** Clicking either entry point puts the UI into a "new chat"
state (empty messages, input box ready). The chat session is only persisted to
the database when the user sends the first message. This avoids polluting the
session list with empty conversations.

### Creating a New Agent

Entry point: "+ Create Agent" button in the Agents sidebar section.

Flow: Immediately creates a new agent record with default values (same defaults
as the current `defaultAgentSpec`), then navigates to its full-width config page
for editing.

### Chat Session List Item Display

Each chat item in the sidebar shows:

- **Title** (primary line)
- **Agent name** (secondary, as a subtle label/badge)
- **Relative timestamp** (`2m ago`, `3h ago`, etc.)
- **Last message preview** (truncated to ~50 chars)

### API Key Handling (v0.1.3 scope)

API keys remain runtime-only in v0.1.3. The key is stored in frontend state and
passed with each message request. The input field lives in the Agent config page
under the Model tab (unchanged from current behavior).

In v0.1.4, API keys will be persisted encrypted (AES-256-GCM) in the `agents`
table on a per-agent basis, eliminating the need to re-enter the key each session.

## Data Model

### New Table: `agents`

```sql
create table agents (
  id text primary key,
  name text not null,
  description text not null,
  spec jsonb not null,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

- `id`: nanoid, primary key.
- `name`: Display name. Not unique (agents distinguished by id).
- `description`: Short description of agent purpose.
- `spec`: Full `AgentSpec` JSON (identity, systemPrompt, model, apps, skills,
  abilities, output).

### Schema Changes: `chat_session`

```sql
alter table chat_session
  add column agent_id text references agents(id);
```

- Nullable initially for migration compatibility; altered to `NOT NULL` after
  backfill completes within the same migration sequence.
- New sessions always set `agent_id`.
- Existing sessions backfilled during migration.

**`agent_spec_snapshot` column:** Stop writing to this column for new sessions
(pass `NULL` or omit). Existing data is retained as a rollback safety net.
The column is NOT dropped in v0.1.3 — scheduled for removal in v0.1.4 once
the migration is confirmed stable. In the shared `ChatSession` type, mark
`agentSpecSnapshot` as optional (`AgentSpec | null`) for this release cycle.

The per-task snapshot (`agent_tasks.agent_spec_snapshot`) remains and continues
to be written — it captures the agent's config at the moment each task is
created, which is the meaningful audit trail under live-spec behavior.

### Data Migration Strategy

Performed inside `runChatMigrationsSequence` (idempotent):

1. Create `agents` table if not exists.
2. If `agents` is empty AND `default_agent_config` has a row:
   - Insert one agent using spec from `default_agent_config`.
   - Use `spec.identity.name` as the agent's `name` and
     `spec.identity.description` as `description`.
3. Add `agent_id` column to `chat_session` if not exists (nullable).
4. Backfill: set `chat_session.agent_id` to the migrated agent for all rows
   where `agent_id is null`.
5. Alter `agent_id` to `NOT NULL` (safe after backfill since all rows now have
   a value).

The `default_agent_config` table is retained (not dropped) as a safety net for
rollback.

## API Surface

### Agent CRUD

| Method | Path                | Description          |
|--------|---------------------|----------------------|
| GET    | /api/agents         | List all agents      |
| GET    | /api/agents/:id     | Get single agent     |
| POST   | /api/agents         | Create new agent     |
| PUT    | /api/agents/:id     | Update agent config  |

No DELETE endpoint in v0.1.3.

### Chat Session Changes

- `POST /api/chat-sessions` now requires `agentId` (instead of full `agentSpec`).
- `POST /api/chat-sessions/:id/messages` fetches agent spec from `agents` table
  using the session's `agent_id` (live spec behavior).
- `GET /api/chat-sessions` response includes `agentId` and `agentName` fields.
- `GET /api/chat-sessions/:id` response includes `agentId` and `agentName`.

### Backward Compatibility

- `GET /api/agent/default` and `PUT /api/agent/default` remain functional
  (read/write the first agent, or fall back to defaults). Deprecated but not
  removed, ensuring the runner and any external consumers still work.

## Scope Boundaries

### In scope (v0.1.3)

- `agents` table + CRUD API
- Sidebar UI restructure (two accordion sections)
- View switching (Chat full-width / Agent config full-width / empty state)
- Live spec on message send
- Chat session ↔ agent binding
- Data migration from `default_agent_config`
- Chat list enhanced display (title, agent name, time, preview)

### Explicitly out of scope

- Agent deletion
- API key encryption/persistence in `agents` table (→ v0.1.4)
- Dropping `chat_session.agent_spec_snapshot` column (→ v0.1.4)
- Chat search / filtering
- Agent duplication / templates
- Multi-user / auth
- Routing (URL-based navigation) — state-driven only for now
