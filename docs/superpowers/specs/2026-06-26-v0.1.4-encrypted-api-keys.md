# v0.1.4: Encrypted API Key Storage

**Status**: Planning  
**Date**: 2026-06-26  
**Version**: 0.1.4

## Overview

Store LLM API keys encrypted at rest on the server side, eliminating the need to send keys with every message. Keys are agent-scoped, encrypted using AES-256-GCM, and decrypted just-in-time before task execution.

## Current State

- API keys are sent from frontend with every message via `runtimeSecrets.apiKey`
- Keys are NOT stored anywhere — they're ephemeral per request
- Keys are redacted in stored messages and logs
- Keys are validated per-request and injected into `AgentSpec.model.apiKey` before running tasks

## Goals

1. **Security**: Encrypt LLM API keys at rest in the database
2. **UX**: Remove the need for users to enter API keys on every chat interaction
3. **Simplicity**: Agent-scoped keys (one key per agent) with straightforward encryption
4. **Safety**: Just-in-time decryption minimizes plaintext exposure in memory

## Non-Goals

- Key rotation support (can be added later if needed)
- Multiple keys per agent or key versioning
- Backward compatibility for existing agents without keys

## Design Decisions

### 1. API Key Scope

**Decision**: Agent-scoped keys

- One encrypted API key per agent
- All chat sessions for an agent use the same key
- Stored in the `agents` table alongside agent metadata

**Why**: Simplifies the model. Users configure credentials once per agent, and all conversations with that agent use the same LLM provider credentials.

### 2. Encryption Method

**Decision**: Symmetric encryption (AES-256-GCM) with a master key

- Algorithm: AES-256-GCM (authenticated encryption)
- Master key stored in environment variable: `LLM_API_KEY_ENCRYPTION_KEY`
- Encryption/decryption handled in Node.js application layer using the `crypto` module
- Each encrypted value includes a random IV (initialization vector) prepended to the ciphertext

**Why**: Standard, portable, no additional PostgreSQL extensions required. The master key lives in the deployment environment alongside other secrets like `DATABASE_URL`.

### 3. Encryption Lifecycle

**Decision**: Encrypt on write, decrypt just-in-time

- **Encryption**: When an agent is created or updated with an API key, encrypt immediately before database write
- **Decryption**: Only decrypt right before calling `runnerClient.runAgentTask()`, not when loading the agent from the database
- **In-memory**: Keep keys encrypted in all in-memory objects and API responses

**Why**: Minimizes the window where plaintext keys exist in memory. Keys are only decrypted at the moment they're needed for LLM API calls.

### 4. API Key Requirement

**Decision**: API keys are required during agent creation

- `apiKey` is a required field in the `POST /api/agents` request body
- Cannot create an agent without providing an API key
- Agents without an `encrypted_api_key` in the database fail validation when attempting to send messages

**Why**: Forces users to complete setup before using an agent. Clear, explicit configuration.

### 5. Database Schema

**Decision**: Add `encrypted_api_key` column to the `agents` table

```sql
ALTER TABLE agents ADD COLUMN encrypted_api_key TEXT;
```

- Column is nullable to allow safe migration
- Runtime validation fails for agents with `encrypted_api_key IS NULL`
- The encrypted value includes the IV prepended to the ciphertext (format: `<iv>:<ciphertext>`)

**Why**: Simple, co-located with agent data. Clear separation between the spec (which is exported/shared) and the encrypted credential (which never leaves the server).

### 6. Migration Strategy

**Decision**: No backward compatibility for existing agents

- Migration adds the `encrypted_api_key` column as nullable
- Existing agents without keys will fail validation when attempting to send messages
- Error message: "Agent API key not configured. Please update the agent settings to add an API key."
- Users must manually update existing agents through the API or UI

**Why**: Simplifies v0.1.4 implementation. No temporary fallback logic. Users are forced to migrate explicitly.

### 7. Frontend UX Changes

**Decision**: Remove API key input from chat interface entirely

- Add API key input field to the "Create Agent" form (required)
- Add API key update field to the agent edit/settings view
- Remove the `apiKey` state and input field from the chat interface
- The `sendChatMessage()` API call no longer includes `runtimeSecrets.apiKey`

**Why**: Cleanest UX. Users configure credentials once during agent creation, then just chat.

### 8. Master Encryption Key Management

**Decision**: Single required environment variable `LLM_API_KEY_ENCRYPTION_KEY`

- Must be set before server startup
- Server refuses to start if the variable is missing or empty
- Clear error message: "LLM_API_KEY_ENCRYPTION_KEY environment variable is required"
- Documentation instructs users to generate a 32-byte random key: `openssl rand -hex 32`

**Why**: Explicit, deliberate key management. Prevents accidental auto-generation that could lead to data loss.

### 9. Key Rotation

**Decision**: No rotation support in v0.1.4

- Single master key, no versioning
- If rotation is needed in the future, write a migration script to decrypt with the old key and re-encrypt with the new key

**Why**: Keep v0.1.4 simple. Complexity of key versioning isn't justified until actually needed.

### 10. Error Handling

**Decision**: Fail immediately on decryption failure

- If decryption fails (corrupted data, wrong master key, etc.), fail the task immediately
- Return clear error to the user: "Failed to decrypt API key for agent"
- Log full error details server-side for debugging
- Do not retry, do not fall back to alternative behavior

**Why**: Decryption failure is a critical error. Making it explicit and visible ensures the problem gets immediate attention.

### 11. API Response Format

**Decision**: Show key presence indicator in API responses

- Add a computed `hasApiKey: boolean` field to agent responses
- Example: `GET /api/agents/:id` returns `{ id, name, spec, hasApiKey: true, ... }`
- Never include the actual key (plaintext or encrypted) in API responses
- Never include masked values like `sk-...***...xyz`

**Why**: Lets the frontend show appropriate UI ("API Key: Configured ✓" vs "API Key: Not Set ⚠️") without exposing any part of the key.

### 12. API Key Updates

**Decision**: Extend existing `PUT /api/agents/:id` endpoint

- Accept optional `apiKey` field in request body
- If `apiKey` is provided (non-empty string), encrypt and update `encrypted_api_key` column
- If `apiKey` is omitted or null, leave existing encrypted key unchanged
- No dedicated key update endpoint

**Why**: Follows REST conventions. Updating an agent's API key is updating agent data — use the agent update endpoint.

### 13. Validation

**Decision**: No API key format validation

- Accept any non-empty string as the API key
- Do not validate against provider-specific patterns (e.g., `sk-...` for OpenAI)
- Let the LLM provider reject invalid keys at runtime

**Why**: Different providers have different formats. Format validation is brittle and breaks when providers change. Runtime errors from the provider are clear and actionable.

### 14. Security Logging

**Decision**: Log operations without any key data

- Log events: "API key encrypted for agent X", "API key updated for agent Y", "Decryption failed for agent Z"
- Include agent ID and timestamp
- Never log the key itself (not plaintext, not encrypted, not masked)
- Use standard application logs — no separate audit table

**Why**: Provides an audit trail for debugging and security review without risking exposure. Keeps logging simple in v0.1.4.

### 15. Database Migration

**Decision**: Add nullable column, fail at runtime if not set

- Migration: `ALTER TABLE agents ADD COLUMN encrypted_api_key TEXT;`
- Column is nullable to allow safe deployment
- At runtime, when sending a message, check if `encrypted_api_key IS NULL`
- If null, return error: "Agent API key not configured. Please update the agent settings."

**Why**: Safest deployment approach. Migration succeeds even if agents exist. Clear error messages guide users to fix their agents.

## Implementation Details

### Encryption Module

Create a new module `apps/api/src/encryption.ts`:

```typescript
import crypto from "crypto";

const ALGORITHM = "aes-256-gcm";
const IV_LENGTH = 16;
const AUTH_TAG_LENGTH = 16;

function getMasterKey(): Buffer {
  const key = process.env.LLM_API_KEY_ENCRYPTION_KEY?.trim();
  if (!key) {
    throw new Error("LLM_API_KEY_ENCRYPTION_KEY environment variable is required");
  }
  // Expect 64 hex characters (32 bytes)
  if (key.length !== 64 || !/^[0-9a-f]{64}$/i.test(key)) {
    throw new Error("LLM_API_KEY_ENCRYPTION_KEY must be a 64-character hex string (32 bytes)");
  }
  return Buffer.from(key, "hex");
}

export function encryptApiKey(plaintext: string): string {
  const masterKey = getMasterKey();
  const iv = crypto.randomBytes(IV_LENGTH);
  const cipher = crypto.createCipheriv(ALGORITHM, masterKey, iv);
  
  const encrypted = Buffer.concat([
    cipher.update(plaintext, "utf8"),
    cipher.final()
  ]);
  
  const authTag = cipher.getAuthTag();
  
  // Format: <iv>:<authTag>:<ciphertext> (all base64)
  return `${iv.toString("base64")}:${authTag.toString("base64")}:${encrypted.toString("base64")}`;
}

export function decryptApiKey(encrypted: string): string {
  const masterKey = getMasterKey();
  const parts = encrypted.split(":");
  
  if (parts.length !== 3) {
    throw new Error("Invalid encrypted API key format");
  }
  
  const iv = Buffer.from(parts[0], "base64");
  const authTag = Buffer.from(parts[1], "base64");
  const ciphertext = Buffer.from(parts[2], "base64");
  
  const decipher = crypto.createDecipheriv(ALGORITHM, masterKey, iv);
  decipher.setAuthTag(authTag);
  
  const decrypted = Buffer.concat([
    decipher.update(ciphertext),
    decipher.final()
  ]);
  
  return decrypted.toString("utf8");
}
```

### PgChatStore Changes

Update `apps/api/src/chat-store.ts`:

```typescript
type AgentRow = {
  id: string;
  name: string;
  description: string;
  spec: AgentSpec;
  encrypted_api_key: string | null; // NEW
  created_at: Date | string;
  updated_at: Date | string;
};

type CreateAgentRequest = {
  spec?: AgentSpec;
  apiKey: string; // NEW: required
};

type UpdateAgentRequest = {
  spec: AgentSpec;
  apiKey?: string; // NEW: optional
};

export class PgChatStore {
  async createAgent(input: CreateAgentRequest): Promise<Agent> {
    // Validate apiKey is non-empty
    const apiKey = input.apiKey?.trim();
    if (!apiKey) {
      throw new Error("API key is required");
    }
    
    const encryptedKey = encryptApiKey(apiKey);
    
    const row = await this.db.query<AgentRow>(
      `INSERT INTO agents (id, name, description, spec, encrypted_api_key, created_at, updated_at)
       VALUES ($1, $2, $3, $4, $5, NOW(), NOW())
       RETURNING *`,
      [nanoid(), name, description, JSON.stringify(spec), encryptedKey]
    );
    
    return {
      ...row.rows[0],
      hasApiKey: true
    };
  }
  
  async updateAgent(id: string, input: UpdateAgentRequest): Promise<Agent> {
    let encryptedKey: string | undefined;
    
    if (input.apiKey !== undefined && input.apiKey !== null) {
      const apiKey = input.apiKey.trim();
      if (apiKey) {
        encryptedKey = encryptApiKey(apiKey);
      }
    }
    
    const updateFields: string[] = ["spec = $2", "updated_at = NOW()"];
    const params: any[] = [id, JSON.stringify(input.spec)];
    
    if (encryptedKey !== undefined) {
      updateFields.push(`encrypted_api_key = $${params.length + 1}`);
      params.push(encryptedKey);
    }
    
    const row = await this.db.query<AgentRow>(
      `UPDATE agents SET ${updateFields.join(", ")}
       WHERE id = $1
       RETURNING *`,
      params
    );
    
    return {
      ...row.rows[0],
      hasApiKey: !!row.rows[0].encrypted_api_key
    };
  }
  
  async getAgent(id: string): Promise<Agent | null> {
    const row = await this.db.query<AgentRow>(
      "SELECT * FROM agents WHERE id = $1",
      [id]
    );
    
    if (!row.rows[0]) return null;
    
    return {
      ...row.rows[0],
      hasApiKey: !!row.rows[0].encrypted_api_key
    };
  }
}
```

## Security Considerations

### Threat Model

**Threats mitigated**:
- Database dump exposure: API keys are encrypted at rest
- Log/message exposure: Keys are redacted before storage (existing behavior preserved)
- Memory inspection window: Keys are only decrypted just-in-time before use

**Threats NOT mitigated**:
- Compromised master encryption key: If `LLM_API_KEY_ENCRYPTION_KEY` is exposed, all stored keys can be decrypted
- Server memory dump during task execution: Decrypted keys exist briefly in memory
- Compromised database with live server access: Attacker with both DB access and ability to query the running server can decrypt keys

### Master Key Protection

**Critical**: The `LLM_API_KEY_ENCRYPTION_KEY` environment variable must be protected with the same rigor as database credentials.

**Best practices**:
- Generate using: `openssl rand -hex 32`
- Store in secrets management system (AWS Secrets Manager, Vault, etc.)
- Never commit to version control
- Rotate if compromised (requires manual migration of encrypted keys)
- Backup securely — losing the key means losing access to all stored API keys

### Logging

**What gets logged**:
- "API key encrypted for agent <id>" (on create/update)
- "API key decrypted for agent <id>" (on task execution)
- "Decryption failed for agent <id>: <error>" (on failure)

**What NEVER gets logged**:
- Plaintext API keys
- Encrypted API keys
- Master encryption key
- Any portion of keys (even masked)

## Troubleshooting

### "LLM_API_KEY_ENCRYPTION_KEY environment variable is required"

The server requires this environment variable to start. Generate one using `openssl rand -hex 32` and set it in your deployment.

### "Agent API key not configured"

This agent was created before v0.1.4 and doesn't have an API key stored. Update the agent via the UI or API to add a key.

### "Failed to decrypt API key for agent"

The master encryption key may have changed, or the encrypted data is corrupted. Check that `LLM_API_KEY_ENCRYPTION_KEY` is set correctly and hasn't changed since the keys were encrypted.
